# Price Feed & Oracle Binding

Two independent price sources cross-validate each other, making price manipulation economically infeasible.

<DocsOracleBindingDiagram />

## Layer 1 — Binance Futures aggTrade

```
wss://fstream.binance.com/ws/{symbol}@aggTrade
```

Symbols: `btcusdt`, `ethusdt`, `solusdt`.

The Binance **Futures** market (not spot) is used because it has tighter spreads, higher liquidity, professional/institutional order flow, and higher notional volume — all of which raise the cost of sustained price manipulation.

The aggTrade stream feeds the OHLC builder (5-second candles). The **settlement candle close** — the OHLC close of the last completed candle at or before `window_end` — is the deterministic reference for win/loss outcomes.

## Layer 2 — Pyth Oracle Checkpoints

For each settlement batch, the ZK circuit samples Pyth at **3 checkpoints**:

| Checkpoint | Timing |
|---|---|
| 1 | `window_start` |
| 2 | `(window_start + window_end) / 2` |
| 3 | `window_end` |

At each checkpoint, the circuit verifies:

$$\text{diff} \times 10{,}000 \le \text{normalizedPythPrice} \times \text{MAX\_DEVIATION\_BPS}$$

where `MAX_DEVIATION_BPS = 200` (2%) and:

$$\text{normalizedPythPrice} = \text{pythPrice} \times 10^{(\text{exponent} + 6)}$$

The `+6` converts from USD to micro-USD to match internal OHLC representation.

Any fabricated OHLC close that deviates more than 2% from the on-chain Pyth price causes proof generation to fail. Because Pyth aggregates from multiple independent sources, an operator can't control both feeds simultaneously.
